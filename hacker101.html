Report

  • Title: Access of unauthorised page
  • Severity: High
  • Description: Can access unauthorised page passing the other action type.
  • Reproduction: The page 4 is not authorised to public but passing the parameter /page/edit/4 reveal the information of the page.
  • Impact: Can reveal the sensitive information to others
  • Mitigation: Apply the authorisation required to edit the page
  • Affected assets: Authorisation

Flag 1

Given hints:

  • Make sure you tamper with every input
  • Have you tested fot the usual culprits? XSS, SQL injection, path injection
  • Bugs often occur when an input should always be one type and turns out to be another
  • Remember, form submissions aren't the only inputs that come from browsers

Report

  • Title: Other type of id is accepted.
  • Severity: None
  • Description: Can pass the id as {id}' which is invalid
  • Reproduction: Pass the parameter in the url after .com as /page/edit/{id}' wher id can be 1,2, or 3 or so on.
  • Impact: Security is breacheable
  • Mitigation: Only valid id can be passed
  • Affected assets: form submission methods

Flag 2

Given hints:

  • Sometimes a given input will affect more than one page
  • The bug you are looking for doesn't exists in the most obvious place this input is shown

Report

  • Title: Input type
  • Severity: High
  • Description: The input type is not defined as a result it can pass html elments
  • Reproduction: Create a new page and pass the input as <button> </button> which will change the list order.
  • Impact: Visibility of the page
  • Mitigation: Enable the type of the input it accepts
  • Affected assets: Elements

Flag 3

Given hints:

  • Script tags are great but what other options do you have?

Report

  • Title: Script of page
  • Severity: High
  • Description: Altering the script might leak imporatant data and give access to threat actors.
  • Reproduction: Adding the onClick attribute to button on page 2 with some function will reveal the flag.
  • Impact: Scrip and server side
